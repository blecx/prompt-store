# Style Guide

## Purpose

This style guide ensures that reports read as professional deliverables rather than ad hoc notes.

## Writing rules

- use precise headings
- use short declarative paragraphs
- define technical terms before using them heavily
- distinguish **finding**, **analysis**, **assumption**, **risk**, and **recommendation** explicitly
- avoid hype language
- avoid empty adjectives such as “powerful”, “innovative”, or “robust” unless demonstrated

## Section logic

A strong section typically contains:

1. purpose of the section
2. what the evidence shows
3. how to interpret it
4. why it matters
5. what should happen next

## Required quality markers

Every substantial report should contain:

- explicit scope
- methodology
- evidence statement
- assumptions
- constraints
- limitations
- recommendation rationale
- impact and trade-off view
- implementation implications

## Tables

Use tables when the reader must compare:

- options
- vendors
- risks
- maturity levels
- findings across categories
- assumptions vs evidence status

## Terms to control carefully

Use these terms consistently:

- finding = directly supported observation
- analysis = interpretation of one or more findings
- recommendation = proposed action based on findings and analysis
- assumption = statement used because complete evidence is unavailable
- limitation = known constraint affecting confidence or scope
- implication = consequence for a decision, design, or operation
